# Post-MVP Infrastructure Plan

## Правило

Этот документ обсуждается и реализуется только после того, как MVP собран и проверен.
Ничего из этого не делается на этапе MVP.

---

## 1. Требования к инфраструктуре

- Все продовые сервисы — только на серверах в РФ.
- Никаких иностранных облаков для продового контура.
- n8n — self-hosted, free community edition, на РФ-сервере.
- LLM-доступ — только через провайдеров, совместимых с требованиями РФ (или локальная модель).
- Данные пользователей не уходят за пределы РФ.

---

## 2. Что нужно определить после MVP

### Сервер
- Провайдер: Yandex Cloud / Selectel / TimeWeb Cloud / VDS у российского хостера
- Минимальная конфигурация для старта: 2 vCPU, 4 GB RAM, 40 GB SSD
- ОС: Ubuntu 22.04 LTS

### База данных
- Тип определяется после MVP: PostgreSQL (предпочтительно) или SQLite для самого простого старта
- Хранит: профили основателей, сигналы рынка, паттерны, карточки возможностей, скоринг-раны
- Бэкап: ежедневный, хранить минимум 7 дней

### n8n
- Self-hosted на том же или отдельном сервере
- Данные n8n (workflows, credentials) хранятся локально, не в облаке
- Credentials — через встроенный vault n8n или переменные окружения сервера

### LLM-доступ
- Вариант A: API совместимого провайдера с серверами в РФ (уточнить список после MVP)
- Вариант B: Локальная open-source модель (Mistral, Llama) через Ollama — если API недоступен
- Fallback всегда должен быть предусмотрен в агентах

---

## 3. Что нужно сделать перед продовым запуском

| Задача | Статус |
|--------|--------|
| Выбрать РФ-хостера | После MVP |
| Настроить сервер и домен | После MVP |
| Установить n8n self-hosted | После MVP |
| Настроить БД и бэкапы | После MVP |
| Настроить SSL (Let's Encrypt) | После MVP |
| Перенести workflow из локального n8n в продовый | После MVP |
| Настроить мониторинг (uptime, ошибки) | После MVP |
| Провести smoke-test на продовом окружении | После MVP |
| Определить LLM-провайдера или локальную модель | После MVP |

---

## 4. Что не делаем даже после MVP (в первой итерации)

- Multi-region setup
- High availability с автофейловером
- Enterprise-grade secret management (Vault, HSM)
- Сложный CI/CD pipeline
- Kubernetes / контейнерная оркестрация
- Мультитенантность с изоляцией данных

Это — если продукт вырастет и появится реальная нагрузка.

---

## 5. Порядок действий

1. MVP проверен, есть первые пользователи или валидация спроса
2. Выбрать хостера в РФ
3. Развернуть n8n self-hosted
4. Импортировать workflow из репо (по инструкции `automation/n8n/docs/import-export.md`)
5. Настроить БД
6. Подключить LLM-провайдера
7. Пройти smoke-tests (`automation/n8n/docs/smoke-tests.md`)
8. Закрыть всё за SSL и базовой аутентификацией
9. Только тогда — открывать доступ пользователям
