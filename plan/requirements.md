# requirements.md

## 1. Бизнес-требования

- Продукт должен помогать пользователю находить сильные бизнес-возможности, а не просто генерировать список идей.
- Продукт должен выдавать ограниченный shortlist возможностей, а не перегружать пользователя большим количеством слабых вариантов.
- Основной целевой рынок продукта на текущем этапе — РФ.
- Продукт должен быть пригоден для запуска как MVP без тяжёлой инфраструктуры.

## 2. Продуктовые требования

MVP должен уметь:
1. Собирать профиль пользователя / основателя
2. Формировать market pain signals
3. Хранить и показывать business patterns / proven patterns
4. Строить Opportunity Card
5. Считать scoring v1
6. Показывать shortlist возможностей
7. Давать recommended entry mode
8. Давать first offer / first test

## 3. Системные требования

- Репозиторий является единственным источником правды.
- Все contracts, правила и сущности должны храниться в repo.
- Всё, что можно сделать детерминированно, должно быть детерминировано.
- LLM должен использоваться как слой анализа, а не как скрытая бизнес-логика.
- Любой важный output LLM должен быть валидируемым.

## 4. Требования к надёжности

- Работа только по chunks.
- После каждого chunk проект остаётся в рабочем состоянии.
- Каждый chunk имеет done criteria.
- После каждого chunk обязателен smoke-check.
- Нельзя трогать сразу несколько крупных областей без отдельного решения.
- Любое изменение должно быть воспроизводимым.

## 5. Требования к сохранности

- Если изменение не сохранено в repo или docs — его не существует.
- Никаких скрытых ручных действий.
- Никаких секретов в repo.
- Все env значения только через .env / secret store / local environment.
- Любая ручная настройка должна быть либо автоматизирована, либо задокументирована.

## 6. Требования к n8n

- Только self-hosted / downloaded / free community.
- Никакой платной n8n cloud.
- На этапе MVP n8n не является центром всей архитектуры.
- n8n подключается после базового MVP-ядра.
- Все workflow хранятся в repo.
- Все workflow имеют:
  - id
  - name
  - purpose
  - input
  - output
  - retry policy
  - error handling
  - owner
- Если нет live access к n8n, запрещено писать, что n8n уже настроен.
- В таком случае допустимо только подготовить repo artifacts:
  - JSON workflows
  - setup docs
  - import/export docs
  - deploy scripts
  - smoke-test instructions

## 7. Требования к агентам

Каждый агент должен иметь:
- id
- purpose
- trigger
- input_schema
- output_schema
- tools
- instructions
- prompts
- fallback
- retry_policy
- error_policy
- linked_workflow

Агенты не должны существовать только как идея "в голове".

## 8. Требования к UI

- Единый стиль: warm dark editorial product UI
- Высокая читаемость
- Минимум визуального шума
- Единый UI kit
- Никаких random style shifts
- Приоритет читаемости над эффектами

## 9. Ограничения на текущем этапе

Сейчас запрещено prematurely fix:
- финальный стек
- продовый сервер
- финальную БД
- домен
- SSL
- продовый CI/CD
- финальный runtime topology

Это обсуждается только после того, как MVP уже собран и проверен.

## 10. Что не строим сейчас

- production hardening
- multi-region setup
- enterprise-grade access model
- финальный billing
- сложные интеграции
- сложную data pipeline инфраструктуру
- full production deployment architecture
